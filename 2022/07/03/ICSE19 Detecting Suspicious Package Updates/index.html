ICSE19 Detecting Suspicious Package Updates

文章很短，数据没有Amalfi有说服力，不做总结了

和Amalfi的区别：

  • 用了k-均值聚类来进行无监督学习，Amalfi使用了监督学习

    • 无监督学习的意思是指数据集没有所谓的标记(比如说是毒包/不是毒包)，k-means聚类通过足够多的数据集学习得到一系列聚类之后，面对新数据时，比对该点和其他聚类的距离是否超过阈值来判断该数据是否正常。

    • 有监督学习的意思就是数据提前存在标签，就是先利用有标签的训练数据学习得到一个模型，然后使用这个模型对新样本进行预测。在本质上，监督学习的目标在于，构建一个由输入到输出的映射，该映射用模型来表示。

    和Amalfi的对比：

    可以看到下方的dependencies和code没有考虑到，但是Amalfi把这部分思想设计在了minified code和binary file，同时由于Amalfi和该文的监测范围不太一样，所以也没有把这两项作为feature

  • 监测范围不同

    • 该文的监测范围在于版本间update中引入的漏洞，因为当时的问题范围就在于update时的漏洞(eslint-scope)
    • Amalfi不止检测了版本间的update，同时还考虑的new package。
    • 对于版本间的feature，Amalfi引入了时间间隔和语义化版本号的类型

  • 对于语义化版本的处理

    • 本文的模型对于不同的语义化版本升级(major.minor.patch)分别进行了聚类学习，并使用elbow method来确定每个模型的k值，分别为1，1，10
    • Amalfi没有考虑对每个语义化升级做不同的模型，而是把版本之间的代码内容的跨度和版本号差距作为feature考虑

后续研究方向

本文结尾指出了一系列研究方向：

  • 把文中模型放大到整个npm生态做评估，来展示大规模数据下的可用性
  • 考虑不同的检测模型设计，比如对每个project检测、每个开发者、或者通过package size来检测，以此来提高precision
  • 建立suspicion的解释：对于模型检测出的suspicious package，提供一定的解释来加快人工审查
  • (对npm站点)针对每天的数据进行看板展示最可疑的包，来提高开发者的安全意识
  • 实现自动化、社交化的方案来帮助开发者发现依赖自动升级的问题
  • 对package registry提供其他的方案，以建立更加安全的package registry
