NDSS21 Towards Measuring Supply Chain Attacks on Package Managers for Interpreted Languages

INTRODUCTION

  • 对第三方库攻击进行了定性分析，并给出了此类攻击在整个软件供应链生态的攻击位置和攻击向量、以及相关人群

    many security researchers are actively investigating attacks on registries and proposing solutions, these approaches seem to be ad-hoc and one-off solutions. A better approach is to understand the extent of the software supply chain abuse and how miscreants are taking advantage of them.

  • 设计了用于第三方库有害包检测的pipeline：MALOSS，其通过三部分(元数据分析、静态代码分析、动态沙盒检测)，加以人工审查来检测有害包

    Using well-known program analysis techniques, we build MALOSS, a custom pipeline tailored for interpreted languages that we use to empirically study the security of package managers.

  • 对当前已经报道和MALOSS检测出的有害包进行了统计分析，给出了针对第三方库有害包攻击的规律，并基于这些规律对相关人群如何规避攻击、优化生态给出了一定的建议。

    We present an in-depth case study to demonstrate the utility of our framework and demonstrate the sophistication of these malicious packages and present their infection vectors, capabilities, and persistence.

    …

    Finally, we propose actionable steps to help improve the overall security of package managers and protect the software supply chain …

Qualitative Analysis

  • Registry Feature

    • Functional：针对PM和Dev，Registry提供的功能
    • Review：RM的审查机制
    • Remediation：补救措施

  • Threat Model

    • 四种身份：RM(Registry Maintainer)、PM(Package Maintainer)、Dev(Developer)、User

    • Attack Vectors：

      • Registry Exploitation【4】：攻击Registry
      • Typosquatting【3】：混淆包名诱导下载
      • Publish【3】：没有混淆包名的投毒上传
      • Account Compromise【3】：获取PM在Registry的权限进行投毒
      • Infrastructure Compromise【2】：黑进PM的开发环境进行投毒
      • Disgruntled Insider【1】：开发者心怀不轨
      • Malicious Contributor【1】：向正常package进行恶意的contribute实现投毒
      • Ownership Transfer【1、3】：把package所有权移交给其他人，在代码层面和包层面都会发生

    • Malicious Behaviors：

      “In supply chain attacks, we consider victims as downstream stakeholders such as Devs and Users.”

      • Stealing：获取敏感信息
      • Backdoor：后门攻击“ leaving a code execution backdoor on victim machines”
      • Sabotage：毁掉受害者的使用环境
      • Cryptojacking：占用受害者机器的资源进行虚拟货币运算
      • Virus：PM/Dev身份不互斥，可以攻击该PM维护的其他包实现病毒传播
      • Malvertising：通过投毒成功的广告展示盈利
      • Proof-of-concept：包本身无毒，但是用来证明某些形式的攻击可以被实现

  • Security Gaps and Broken Trust

MALOSS

Metadata Analysis

对package相关的概要信息进行分析（directly retrieved from registry APIs）

Static Analysis

The static analysis focuses on analyzing source files of the corresponding interpreted language for each package manager and skips embedded binaries and native extensions.

包含了人工API标记和可疑API使用分析

  • 人工API标记

    we focus on four types of runtime APIs in the static analysis, namely, network, filesystem, process, and code generation.

  • API使用分析

    使用AST进行分析，对每个包先生成依赖树，再从叶子节点往上集成、分析，以此来保证每个package只被分析一次

    We parse source files of packages into Abstract Syntax Trees (AST) using state-of-the-art libraries [42]–[45] and search for usage of manually labeled APIs in AST.

    to increase efficiency and reduce failures, we perform modularized API usage analysis which analyzes each package only once.

    We first build a dependency tree of all packages and analyze API usage for ones without dependencies. We then walk up the dependency tree and combine APIs of packages and their dependencies.

    To perform dataflow analysis, we survey and test open-source tools for each interpreted language and choose PyT for Python, JSPrime for JavaScript and Brakeman for Ruby.

    We adapt these tools to analyze packages with a customized configuration of sources and sinks, and output identified flows between any source-sink pair.

Dynamic Analysis

通过沙盒环境能够把包中的二进制文件和本地扩展也进行分析，同时能够检查package内置功能函数和类的表现。

In comparison to static analysis, dynamic analysis considers source files, as well as embedded binaries and native extensions, but it does not have visibility into the runtime environment.

The analysis consists of two parts, package execution within Docker containers for sandboxing and dynamic tracing using Sysdig for efficiency and usability.

  • Package Execution

    分为了install, embedded binary, import and functional.

    • install：运行安装脚本
    • embedded binary： run executables
    • import：通过import语句触发初始化脚本
    • functional：对方法和类进行模糊测试(fuzz)

    The current prototype invokes exported functions, initializes classes with null arguments, and recursively invokes callable attributes of modules and objects.

  • Dynamic Tracing

    After cross-comparison, we choose Sysdig as the tracing tool due to its high efficiency and good usability.

    While prototyping, we track system calls related to IPs, DNS queries, files, and processes and dump them into files to allow further processing.

True Positive Verification

在之前三种检查得到可疑包之后对包的处理：

  • 如果满足heuristic rules，自动报告为毒包
  • 若不满足，则进行人工检查，如果是毒包，则更新heuristic rule set规则集并人工报告为毒包

FINDINGS & MITIGATION

FINDINGS

实验环境主要使用了20个本地工作站，硬件环境为64GB内存和8个3.60GHz的Intel Xeon cpu，操作系统为Ubuntu 16.04。

在检查完PyPI、npm、RubyGems中的一百万个包后，MALOSS发现了339个有害包，上报给RM之后其中82%被移除，同时值得提出339个包中有三个包(paranoid2, simple_captcha2, datagrid)拥有着超过10万的下载量，表示第三方包攻击中有着大量的受害者。作者团队把这三个包的都标记了CVE漏洞号以便于查验。

针对报道中提到的312个有害包和MALOSS扫描到的339个有害包，总结了一些发现：

  • 第三方包通过直接依赖其他包会存在着许多的间接依赖
  • typosquatting和account compromise是最常见的两种攻击向量
  • stealing和backdoor是最常见的有害表现
  • 20%的有害包在package manager中保存了超过400天，并且有着超过1k的下载量
  • 被动DNS获取到的数据说明了：软件供应链攻击的有效性，以及更大用户基数能够帮助更有效地修复漏洞
  • 存在更加高级的攻击方式：文中提到code obfuscation、multi-stage payload、logic bomb、non-latest release
  • 第三方包管理中心缺乏良好定义的规章制度，从而导致了information stealing和user tracking(用户跟踪)的界限不明确

MITIGATION

  • MALOSS团队给出的一些建议

    • Registry Maintainers：加强账号认证/Typo拦截/规范化Ownership Transfer/对可疑包引入crowd-sourcing manual reviews(比如公众投票)/增加毒包相关使用人群的通知机制

    • Package Maintainers：保护好账号/注意contributor的恶意行为/加入RM的review机制

    • Developers：制作私有源收集安全包/定时检查security advisories/安装包之前进行一定的检查/加入RM的review机制

    • End-users：加强安全意识(使用官方资源、有名气的站点)

  • MALOSS本身存在的一些限制

    • 仅在Ubuntu16.04进行动态分析，没有考虑其他运行环境
    • 静态分析由于dynamic type而不准确
    • 动态检查方式有限，还可以引入不同运行环境、force-execution、symbolic execution等
    • 攻击者的方法也在进化
    • heuristics rules有限，存在false negative无法被人工审查，true positive的人工审查也会消耗资源
