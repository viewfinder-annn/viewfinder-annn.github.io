NDSS21 Towards Measuring Supply Chain Attacks on Package Managers for Interpreted Languages

INTRODUCTION

  • 对第三方库攻击进行了定性分析，并给出了此类攻击在整个软件供应链生态的攻击位置和攻击向量、以及相关人群

    many security researchers are actively investigating attacks on registries and proposing solutions, these approaches seem to be ad-hoc and one-off solutions. A better approach is to understand the extent of the software supply chain abuse and how miscreants are taking advantage of them.

  • 设计了用于第三方库有害包检测的pipeline：MALOSS，其通过三部分(元数据分析、静态代码分析、动态沙盒检测)，加以人工审查来检测有害包

    Using well-known program analysis techniques, we build MALOSS, a custom pipeline tailored for interpreted languages that we use to empirically study the security of package managers.

  • 对当前已经报道和MALOSS检测出的有害包进行了统计分析，给出了针对第三方库有害包攻击的规律，并基于这些规律对相关人群如何规避攻击、优化生态给出了一定的建议。

    We present an in-depth case study to demonstrate the utility of our framework and demonstrate the sophistication of these malicious packages and present their infection vectors, capabilities, and persistence.

    …

    Finally, we propose actionable steps to help improve the overall security of package managers and protect the software supply chain …

Qualitative Analysis

  • Registry Feature

    • Functional：针对PM和Dev，Registry提供的功能
    • Review：RM的审查机制
    • Remediation：补救措施

  • Threat Model

    • 四种身份：RM(Registry Maintainer)、PM(Package Maintainer)、Dev(Developer)、User

    • Attack Vectors：

      • Registry Exploitation【4】：攻击Registry
      • Typosquatting【3】：混淆包名诱导下载
      • Publish【3】：没有混淆包名的投毒上传
      • Account Compromise【3】：获取PM在Registry的权限进行投毒
      • Infrastructure Compromise【2】：黑进PM的开发环境进行投毒
      • Disgruntled Insider【1】：开发者心怀不轨
      • Malicious Contributor【1】：向正常package进行恶意的contribute实现投毒
      • Ownership Transfer【1、3】：把package所有权移交给其他人，在代码层面和包层面都会发生

    • Malicious Behaviors：

      “In supply chain attacks, we consider victims as downstream stakeholders such as Devs and Users.”

      • Stealing：获取敏感信息
      • Backdoor：后门攻击“ leaving a code execution backdoor on victim machines”
      • Sabotage：毁掉受害者的使用环境
      • Cryptojacking：占用受害者机器的资源进行虚拟货币运算
      • Virus：PM/Dev身份不互斥，可以攻击该PM维护的其他包实现病毒传播
      • Malvertising：通过投毒成功的广告展示盈利
      • Proof-of-concept：包本身无毒，但是用来证明某些形式的攻击可以被实现

MALOSS

Metadata Analysis

对package相关的概要信息进行分析（directly retrieved from registry APIs）

Static Analysis

The static analysis focuses on analyzing source files of the corresponding interpreted language for each package manager and skips embedded binaries and native extensions.

包含了人工API标记和可疑API使用分析

  • 人工API标记

    we focus on four types of runtime APIs in the static analysis, namely, network, filesystem, process, and code generation.

  • API使用分析

    使用AST进行分析，对每个包先生成依赖树，再从叶子节点往上集成、分析，以此来保证每个package只被分析一次

    We parse source files of packages into Abstract Syntax Trees (AST) using state-of-the-art libraries [42]–[45] and search for usage of manually labeled APIs in AST.

    to increase efficiency and reduce failures, we perform modularized API usage analysis which analyzes each package only once.

    We first build a dependency tree of all packages and analyze API usage for ones without dependencies. We then walk up the dependency tree and combine APIs of packages and their dependencies.

    To perform dataflow analysis, we survey and test open-source tools for each interpreted language and choose PyT for Python, JSPrime for JavaScript and Brakeman for Ruby.

    We adapt these tools to analyze packages with a customized configuration of sources and sinks, and output identified flows between any source-sink pair.

Dynamic Analysis

通过沙盒环境能够把包中的二进制文件和本地扩展也进行分析，同时能够检查package内置功能函数和类的表现。

In comparison to static analysis, dynamic analysis considers source files, as well as embedded binaries and native extensions, but it does not have visibility into the runtime environment.

The analysis consists of two parts, package execution within Docker containers for sandboxing and dynamic tracing using Sysdig for efficiency and usability.

  • Package Execution

    分为了install, embedded binary, import and functional.

    • install：运行安装脚本
    • embedded binary： run executables
    • import：通过import语句触发初始化脚本
    • functional：对方法和类进行模糊测试(fuzz)

    The current prototype invokes exported functions, initializes classes with null arguments, and recursively invokes callable attributes of modules and objects.

  • Dynamic Tracing

    After cross-comparison, we choose Sysdig as the tracing tool due to its high efficiency and good usability.

    While prototyping, we track system calls related to IPs, DNS queries, files, and processes and dump them into files to allow further processing.

True Positive Verification

在之前三种检查得到可疑包之后对包的处理：

  • 如果满足heuristic rules，自动报告为毒包
  • 若不满足，则进行人工检查，如果是毒包，则更新heuristic rule set规则集并人工报告为毒包

FINDINGS & MITIGATION

FINDINGS

实验环境主要使用了20个本地工作站，硬件环境为64GB内存和8个3.60GHz的Intel Xeon cpu，操作系统为Ubuntu 16.04。

在检查完PyPI、npm、RubyGems中的一百万个包后，MALOSS发现了339个有害包，上报给RM之后其中82%被移除，同时值得提出339个包中有三个包(paranoid2, simple_captcha2, datagrid)拥有着超过10万的下载量，表示第三方包攻击中有着大量的受害者。作者团队把这三个包的都标记了CVE漏洞号以便于查验。

针对报道中提到的312个有害包和MALOSS扫描到的339个有害包，总结了一些发现：

  • 第三方包通过直接依赖其他包会存在着许多的间接依赖
  • typosquatting和account compromise是最常见的两种攻击向量
  • stealing和backdoor是最常见的有害表现
  • 20%的有害包在package manager中保存了超过400天，并且有着超过1k的下载量
  • 被动DNS获取到的数据说明了：软件供应链攻击的有效性，以及更大用户基数能够帮助更有效地修复漏洞
  • 存在更加高级的攻击方式：文中提到code obfuscation、multi-stage payload、logic bomb、non-latest release
  • 第三方包管理中心缺乏良好定义的规章制度，从而导致了information stealing和user tracking(用户跟踪)的界限不明确

MITIGATION

  • MALOSS团队给出的一些建议

    • Registry Maintainers：加强账号认证/Typo拦截/规范化Ownership Transfer/对可疑包引入crowd-sourcing manual reviews(比如公众投票)/增加毒包相关使用人群的通知机制

    • Package Maintainers：保护好账号/注意contributor的恶意行为/加入RM的review机制

    • Developers：制作私有源收集安全包/定时检查security advisories/安装包之前进行一定的检查/加入RM的review机制

    • End-users：加强安全意识(使用官方资源、有名气的站点)

  • MALOSS本身存在的一些限制

    • 仅在Ubuntu16.04进行动态分析，没有考虑其他运行环境
    • 静态分析由于dynamic type而不准确
    • 动态检查方式有限，还可以引入不同运行环境、force-execution、symbolic execution等
    • 攻击者的方法也在进化
    • heuristics rules有限，存在false negative无法被人工审查，true positive的人工审查也会消耗资源

ICSE22 Practical Automated Detection of Malicious npm Packages

Introduction

截至2021年9月，npm包管理站点拥有超过1.7百万第三方包，同时每周开发者会上传超过10万个包版本，这其中包括了新package和已有package的新版本。

发布新包/新版本是相同的流程，任何一个得到验证的npm站点用户上传包之后自然而然的成为该包的maintainer，永和可以通过对包目录上传新tarball文件来随时上传新版本。maintainer必须保证在包发布之前进行一定的构建(build)工作。包管理站点仅仅保存这些tarball而对包里的内容大部分是不可知的。npm过人的欢迎程度使得其被攻击者利用来发布有害package，进行有害的活动。(比如eslint-scope和event-stream等)

想要从根本上解决包管理站点投毒问题需要对npm站点本身做出巨大的改变，比如更加严格的包上传机制，或者对第三方包实现访问控制来降低投毒包的伤害。

作者团队在文中的目标更小，不需要对npm站点做出任何变化，实现了尽可能快的包检测工具，并把检测到的毒包报告给了审查人员来进行毒包移除。

对于这种工具作者团队给出了三个关键词：automated、efficient、accurate，并基于三种目标提出了Amalfi模型，其包含以下三种成分：

  • machine-learning classifiers trained on labelled examples of malicious and benign packages
  • a reproducer that rebuilds a package from source and compares the result with the version published in the registry
  • a clone detector that finds (near-)verbatim copies of known malicious packages

对于分类器，作者团队使用了三种不同的策略：decision trees, Naive Bayesian classifiers, and one-class SVMs，同时使用再构建技术在一定程度上减少false positive，使用逐字符(a simple clone detector that hashes the contents of a package (minus the package name and version, which are always unique))的克隆检测来减少false negative。

作者团队对于Amalfi开展了两类实验，一类实验基于实时大量包数据，最终在一周新上传的共96287个包中检测出95个有害包；另一类实验选择MALOSS模型检测出的数据进行交叉验证观测了Amalfi的precision和recall。

论文贡献：

  • 实现了Amalfi，并在实验过程中检测出了95个有害包，在三种分类策略的选择中决策树占优，但其他方法也有其贡献。
  • Amalfi效率不俗，只需要数秒内就能够提取一个packge的feature并运行分类器，同时retraining成本低廉，可以不断地对模型进行优化
  • 假阳性率一开始很高，但在分类器训练更多数据之后，假阳性率有效降低了。同时交叉验证中说明决策树能够达到40%的recall，说明假阴性率是可以接受的。

While the building blocks we use have been proposed before, the novelty of our approach lies in their combination, and a more thorough exploration and evaluation of the design space.

项目地址：githubnext/amalfi-artifact

Our Approach

Feature Set

Amalfi的分类器基于11个feature建立：9个单版本feature和2个跨版本feature

  • 单版本feature：

    1. 访问个人信息的行为(personally-identifying information，比如信用卡号、密码、cookies等)

    2. 访问特定的系统资源

      a. 访问文件系统：读写文件

      b. 新进程创建

      c. 网络资源访问：发送/接收数据

    3. 使用特定API

      a. 加密解密API

      b. 数据编码API(比如encodeURIComponent等)

      c. 动态代码生成和执行API(比如eval、Function等)

    4. 使用包安装脚本

    5. 包含极简化代码(minified code，从JavaScript源代码中删除所有不必要的字符而不改变其功能的过程。这包括删除空格、注释和分号，以及使用较短的变量名和函数)，或者二进制可执行文件

  • 跨版本feature

    1. 和上一个版本的时间跨度太短或者太长(可能是账号盗取)
      • 对于new package，作者团队把和上一个版本的时间跨度置为0来不遗漏feature
    2. 和上一个版本的语义化版本号差距与代码内容不符合(major, minor, patch, build, or pre-release)

Feature Extraction

Amalfi使用了tree-sitter来提取代码的AST从而寻找对应feature的句法结构，比如包含着password的字符常量、对fs的import、eval和Function的调用等。

为了检查包安装脚本，Amalf检查package.json来查找preinstall、install、postinstall的定义和内容。

对于minified code和binary，Amalfi计算了包中所有文件的香农熵，通过所有文件的平均值和标准差进行检查。

对于的跨版本的feature，版本时间使用了npm view time来提取，语义化版本号则是使用现成的语义化版本号库来检查update type，并寻找上一个版本号的包，通过单版本feature的提取来比对。

Classifier training

基础数据集：作者团队向npm站点获取了643个截至2021.07.29检测出并归档的有害包，其中63个包是无害包的有害版本。加上1147个同一天上传的正常包，一共1790个包。基础数据集使用了过采样技术(现实情形下有害包的比例要小得多)

值得指出通过劫取已有包的账号来投毒会有更加大的影响，但这种攻击很少很少发生。但直接上传毒包和这种账号劫取攻击的内容和形式是相同的，所以Amalfi都能进行检测。

由于imbalanced data，同时需要算法去分析哪些feature更加重要，我们选择了decision trees, Naive Bayesian classifiers, and One-class Support Vector Machines (SVMs)三种学习算法。决策时有能力解释每种feature对最终决策的影响力，同时后两种算法针对不均匀的数据集的异常检测有着很好的表现。

  • classifier training：sklearn library
  • decision tree：use information gain as the split criterion
  • Naive Bayesian classifiers：使用Bernoulli变体，它只能处理布尔特征，所以我们省略了离散特征（熵的平均值和标准差以及更新时间），并将其他特征折叠为1，如果该特征存在，则为0
  • SVM：选择了一个线性核，并只对良性例子进行训练，因为这个分类器的任务是检测与良性例子明显不同的离群版本。同时使用留一法(leave-one-out experiment)来决定the 𝜈 parameter of the SVM，最终得到𝜈值为0.001，说明该分类器期待从1000个包中发现一个有害包。

Reproducer and clone detector

reproducer：是一个启发式的过程，在package信息中获取url，package version也能确定commit的git SHA。但是由于1)许多包的开发流程不规范，找不到正确的commit，2)build command也没有事先规定，3)build tools的版本也缺少信息。这使得reproducer的成功率很低，但它作为false-positive filter还是有其作用。

clone detector：计算包中内容的MD5哈希值，同一系列已知有害包的哈希值比对，在计算哈希的时候省略了package.json文件来消除版本的影响。但克隆检测没有使用任何模糊匹配策略，导致只有逐字符的测龙才会被检测到。

Evaluation

  • RQ1 Does Amalfi find malicious packages in practice?

  • RQ2 Is it accurate enough to be useful?

  • RQ3 Is training and classification fast enough to be useable?

为了回答三个问题，Amalfi开展了两个实验：对新包的检测验证其有效性/对已知有害包的检测验证其precision和recall

Experiment 1: Classifying newly published packages

选择了2021.07.29到2021.08.04一周内的新包进行检测。

通过三种classifier和clone detector四个检测器挑出可能的有害包，再使用reproducer来消除一定的FP。

对于剩下挑出的包，两位作者进行人工审查来确定其是否有害，每个作者审查一半的包，对于一个作者发现的有害包另一个作者同样进行再审查。

每天审查结束之后，把当日的数据作为训练集输入继续进行训练来提高其表现(虽然没有考虑到FN，但现实情况也无法把所有FN都摘出)。

Experiment 2: Classifying labelled data

作者团队使用了MALOSS检测出的npm有害包来进行验证。由于数据集过小，无法实施train-and-test方法，最终采用了10折交叉验证来评估。由于数据不均匀，作者团队使用了分层采样来保证每折的有害包/良包分布。

通过剔除MALOSS中和基础数据集重叠的有害包，类似地填充了一部分良包后，得到了总共372个包的数据集，其中有着40个有害包。

RQ1: Practical performance on newly published packages

实验一中三种分类器数据后面的加减：

TP中的+：在检测出TP之后基于这些TP进行clone detect从而发现的新有害包

FP中的-：在检测出FP之后，对于每个FP进行reproduce从而发现的良包

实验一的发现：

  • 每天新发布的包数量很多，但也不均匀(比如07.29和08.01的差距，其中前者是一个周四，后者是一个周六)
  • Amalfi能够检测出有害包，三种分类器中决策树表现最好，但其他分类器也有其贡献。最终团队发现了95个有害包(报告给npm团队后均被删除)，相比于之前检查到的643个有害包而言数量非常可观。
  • 在第一天三种分类器的FP率都很高，但后续的几天里明显降低。说明了初始数据集存在一定的缺陷，但Amalfi经过训练之后表现能够明显提升。
  • reproducer有着很低的成功率，类似地clone detector成功率也不高，08.04的clone成功率存在异常，但这17个TP和决策树检测出的大部分重合。

RQ1的回答：we can answer RQ1 in the affirmative: Amalfi does indeed detect malicious packages in practice. Further, since all the packages found by Amalfi and reported to npm had not been identified before, we can confidently claim that our approach complements existing solutions for malicious package detection in npm.

RQ2: Accuracy

实验二的第一行表示对基础数据集的验证，由于SVM的𝜈值是基于该数据集，所以最终表现打了括号。

第二行表示对MALOSS提取出的数据的验证。可以看到recall高于第一行，但precision更低。说明这两种指标存在着trade-off关系，同时还要注意到MALOSS团队报告给npm团队的有害包有一部分并没有被下架。

针对Amalfi和MALOSS更加细致的比较无法实现，因为MALOSS的庞大pipeline，难以把基础数据集适应到MALOSS模型，同时开销过大。

RQ2的回答：we can give a cautiously positive answer to RQ2: Amalfi is reasonably precise and does not produce an overwhelming number of results, making manual triaging of results by a human auditor feasible. The second experiment suggests that there may be a good number of false negatives, but at least the numbers for the decision tree look promising.

RQ3: Performance

针对效率作者团队观测了以下三个指标：(1)训练分类器的时间，(2)提取单个包feature的时间，(3)分类一个包的时间

训练时间结果：

feature提取时间：By and large, feature extraction takes less than ten seconds, and for over half the packages considered it takes less than one second.

分类时间： For all the classifiers, the time for prediction was less than a second.

RQ3的回答：we can give a positive answer to RQ3: Amalfi is fast enough for practical use.

Threats to validity

  • 基础数据集中存在着克隆内容的有害包
  • 对于Amalfi检查出的每个包，人工审查的时间限制在了5分钟，可能漏掉TP，但FP不一定漏掉，毕竟所有上报给npm的包都被下架。
  • 没有被Amalfi检查出的包默认为良包，并不是一个完全正确的预想

Discussion

每一个检测出的有害包都使用了安装脚本，有一些传输了个人信息，有一些没有(可能是作为proof-of-concept的包出现)

图三说明了有害包出现的feature

同时我们的feature extractor没能够提取出个人信息获取的feature，但由于其他feature的存在能够检查出有害包，说明系统有其健壮性，一些没有被检测到的feature并不代表着没有作用。

major update是最常见的更新，这与“有害包的注入通常和minor update相关”的直觉相违背，这也可能说明着存在更多没有检查出来的有害包。

图五说明了有害包和良包的香农熵统计和时间间隔，红线代表着中位数。可以看到有害包的熵明显高于良包(4.69 vs 0.001)，同时更新时间间隔短于良包(7.02s vs 2217.18s)

图六展示了三种分类器的表现，可以看到决策树占大头，但其他分类器也能分辨出决策树没有发现的有害包，所以三种分类器一起使用效果更好。

对于feature的选择，实验一中07.29-07.31都没有涉及到cryptographic functionality，但在08.01之后，所有11种feature都有考虑到了，说明特征集没有冗余。

最终团队还尝试了随机森林算法代替决策树，最终效果并不理想，同时对于SVM的布尔化配置也进行了调整，但会造成更坏的结果。

Conclusion

作者团队提出了Amalfi，一个自动检测npm有害包的工具。其基于机器学习的分类器，再构建器和克隆检测器。对于分类器采用了决策树，朴素贝叶斯分类器和单类支持向量机。其中第一种表现最好但另外两种对检测结果也有补充。

  • 更深程度的feature extraction，不再是纯句法分析，可以引入部分程序分析的工具。
  • 更加有效的clone detection
  • 如何结合不同的分类器，比如ranking系统来帮助人工审查
  • 迁移到更多的package registry

ICSE19 Detecting Suspicious Package Updates

文章很短，数据没有Amalfi有说服力，不做总结了

和Amalfi的区别：

  • 用了k-均值聚类来进行无监督学习，Amalfi使用了监督学习

    • 无监督学习的意思是指数据集没有所谓的标记(比如说是毒包/不是毒包)，k-means聚类通过足够多的数据集学习得到一系列聚类之后，面对新数据时，比对该点和其他聚类的距离是否超过阈值来判断该数据是否正常。

    • 有监督学习的意思就是数据提前存在标签，就是先利用有标签的训练数据学习得到一个模型，然后使用这个模型对新样本进行预测。在本质上，监督学习的目标在于，构建一个由输入到输出的映射，该映射用模型来表示。

  • 监测范围不同

    • 该文的监测范围在于版本间update中引入的漏洞，因为当时的问题范围就在于update时的漏洞(eslint-scope)

    • Amalfi没有考虑对每个语义化升级做不同的模型，而是把版本之间的代码内容的跨度和版本号差距作为feature考虑

FSE21 LastPyMile-identifying the discrepancy between sources and packages

Introduction

​ 当下FOSS(Free and Open Source Software)，被广泛使用，99%代码仓库含有开源代码。

​ 对于第三方组件市场(Npm、PyPI、RubyGems)的package，开发者总会下载其**预先构建(pre-built)*好的包进行使用。但这是在code repository中的源代码到构建好的包的最后一公里不会引入差异(discrepancies)*的前提下，才会认为它没有安全风险。

​ 但在package从源代码到构建好的包中总会引入差异，这种差异可能是由于自动代码生成工具/未在code repository记录的更改造成的，但也可能会是恶意植入的差异代码。

​ 可重现(Reproducible)构建是理想的解决方案，但实际上不可行，因为构建时会引入许多环境相关的信息(比如时间戳、版本、git tag相关信息等)

​ 当前的PyPI包上传审查机制仅仅检查setup.py，并且会出现许多假阳性报告。

​ 由于对于第三方组件市场的审查资源有限(the administrators had to evaluate 1874 new updates per day, with an average of 3500 files generated by more than 76 997 developers)，在审查时不能引入太多的阳性信号，开销过大。

​ 据此作者团队认为关键检查在于code injection attacks，通过检查最终制品和source code的区别就能有效地缩小审查范围，并且还能够结合之前的审查工具进行更加有效的审查。

Research Questions

RQ1 Can we effectively and efficiently identify differences?

git log能够实现package和源码仓库的比对，但为了拿到所有信息需要把git日志中所有信息获取到，开销过大。

lastpymile团队通过比对package和源码仓库每个文件的哈希值对比，来筛选差异文件和差异行。

RQ2 How big are the ‘normal’ differences between source code and package repositories?

Differences are too many (65% of artifacts and 22% of files in our sample) and too diverse for reproducible builds to be a solution. Yet, only few modifications happen in Python source files (2.6% of files) so that vetting might be a feasible alternative.

作者团队发现差异广泛存在，但对于源码角度，仅有2.6%的文件存在着差异，所以进行比对之后再审查能够大大降低需要审查的内容。

RQ3 Can LastPyMile be combined with package scanners while keeping the number of alerts manageable by a human?

LastPyMile可以同其他扫描工具配合使用，如下图，作为其他几种(Malware Checks、Bandit Checks等)的辅助来降低审查范围。

Findings

LastPyMile通过首先收集文件哈希值来进行差异比对，之后再对于哈希值不同的文件进行line-by-line的比对得到phantom files and lines的输出。

数据范围来自于PyPI最受欢迎的4000个包，最终筛选出93252个包制品和2438个包。

总体分析

  • 可以看到差异现象非常普遍，但在源码级别差异的数量是可以接受的(占5.8%的软件制品和2.6%的文件中)。

文件名和API调用统计

团队还统计了Python文件中出现差异的文件名频数，以及差异文件中API频数统计。

  • 可以看到__init__.py和setup.py是差异最多的文件，因为构建工具会注入很多额外信息(时间戳、版本信息等)，并且和Git Tag相关的version文件也排在前列。

  • 对于差异内容的API调用，存在许多的网络相关的API，但这些不一定是有害的，比如说请求下载其他依赖这种行为也会引起这些API的调用。

结合其他工具的效果

作者团队把LastPyMile和其他扫描工具进行了结合(PyPA自带的Warehouse Malware Checks和Bandit工具)，通过对三个流行包和三个对应的投毒包进行了扫描(使用LastPyMile/不使用LastPyMile)

  • 仅仅对setup.py进行扫描的覆盖率是很低的，从代码覆盖率和不使用LastPyMile的Alert数目都能看出来。
  • 不使用LastPyMile的情况下，不管是setup.py还是整个包的审查，都会出现Alert，对于正常包而言明显是假阳性现象。
  • 不使用LastPyMile，可以看到正常包的Alert会比毒包更多，这是由于正统包往往会有更广泛的代码内容。
  • 使用LastPyMile之后，仅仅审查差异代码内容，可以看到Alert数目大大减少，并且正常包不再出现Alert，毒包出现了Alert。

THREATS TO VALIDITY

  • only consider repositories hosted on GitHub
  • focuses on the Python packages in PyPI, only considered the top 4000 packages hosted on PyPI, out of more than 250 000 packages
  • LastPyMile checks only the code absent from source repositories even though malicious code could also be included in the versioned code, either directly or in tests
  • We limit the line-by-line analysis to files with file extension .py
  • Other phantom files might be also be used to inject malware(比如test directories和the upload of modules specific to a developer development environment)
  • PyPI packages contain other executables
  • We only check additions of code lines in the present version
  • Some packages contain code automatically generated by tools like Swagger Codegen or Python distutils

Conclusion

当下的PyPI包检查器仅仅检查包制品，这种方式会导致很多假警告。同时扫描整个package会是一项效率低下的工作。LastPyMile通过筛选出包制品和源码仓库不同的.py代码内容，结合其他漏洞扫描工具，有效地降低了第三方组件库的供应链攻击的审查成本和时间。
